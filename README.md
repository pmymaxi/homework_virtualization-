## Домашнее задание к занятию 1. «Введение в виртуализацию»
### Ответ задачи №2
**1. Высоконагруженная база MySql, которая критична к отказу.**

Первый критерий, который явно указан это “критична к отказу”, значит необходимо рассматривать модель с основным параметром HA (High Available). Тут можно построить и на паравиртулизации и на физическом сервере. Мой выбор, на платформе паравиртулизации. Метод реализации HA, кластер в кластере. Кластеризация на уровне гипервизоров и кластеризация на уровне виртуальных машин (Guest cluster). Отказоустойчивость выполняется на уровне гипервизоров и отказоустойчивость приложений выполняется на уровне виртуальных машин. Производительность при использовании платформы паравиртулизации не значительно будет проигрывать в производительности физического сервера. Основным горлышком может стать диск, так как MySql требовательный к операциям ввода/вывода (I/O), улучшается на уровне железа (использование высокопроизводительных дисков). Так как база высоконагруженная, использовал балансировщик нагрузки не только по network, но и по нагрузки VM.  

**2. Платформа для web-приложений**

Выбор на платформе виртуализации уровня OS, так как это контейнеризация. Одна из причин выбора, это масштабируемость (например, в случае нагрузки), быстрый deploy, запуск, потребление ресурсов (минимальное по сравнению если приложение разворачивалось на VM со своей гостевой OS).

**3. Windows-системы для использования бухгалтерским отделом.**

Инфраструктура на базе Windows, то однозначно использовал паравиртуализацию (Hyper-V), потому что семейство Windows, а это совместимость, автоматизация, интеграция c продуктами и сервисами этого семейства.

**4. Cистемы, выполняющие высокопроизводительные расчёты на GPU**

Однозначно на физическом сервере, потаму что, происходит выполнения очень объемных и высоконагруженные процессов, с участием микроядер GPU, в таком случае дополнительная абстракция существенно будет замедлять и нагружать процесс обработки. 

### Ответ задачи №3
#### Сценарий №1	
Так как инфраструктура построена на базе ОС Linux и Windows, но преимущественно Windows, то тогда выбор гипервизора Hyper-V с системой управления SCVMM (System Center Virtual Machine Manager). 
Во-первых, большая часть инфраструктуры на Windows, а это, интеграция с сервисами и продуктами Windows. Во-вторых Hyper-V имеет поддержку OS Linux. 
Управление гипервизором можно организовывать как на SCVMM так и с использованием встроенных компонентов кластеризации, Windows Admin Center, powershell (управление и выполнение автоматизаций по custom сценариям/шаблонам) или в урезанном функционале консоль Hyper-V management.
Балансировку нагрузки можно выполнить по нескольким уровням:
- Балансировка на уровне гипервизора при помощи функции Dynamic Load Balancing механизма Hyper-V Load Balancing (CPU, RAM, NETWORK между Node кластара), выполняется через роль Failover Cluster совместно с использованием SCVMM. Если Nodе1 перегружена выполняет live-migration VM на Node2.
- Балансировка на уровне Network Hyper-V Virtual Switch. 
  
Репликация данных Hyper-V Replica на Node.

Механизм создание резервных копий на Hyper-V можно организовать штатными средствами (Windows Server Backup), выполнение backup VM и OS через планировщик, но лучше использовать специальные средства резервных копий, например Veeam Backup & Replication, можно использовать его вместо штатного Replica.
SCVMM обеспечит функционалом управления кластерами, автоматическим развертыванием ВМ так с использованием шаблона VM, в общем одним словом даст нам оркестрацию VM.
#### Сценарий №2	
Open source решение Proxmox VE, на борту KVM который работает непосредственно в ядре Linux тем самым по производительности близок к аппаратному исполнению. Также поддержка Windows ОС (правда можно сказать низкоуровневая из-за процесса добавления драйверов virtio).  
#### Сценарий №3	
Windows Hyper-V Server в редакции core или ESXI (но без возможности оркестрации). Hyper-V Server core уже имеется компонент кластеризации, управление WMI, через консоль Hyper-V management, powershell (но с предустановленными модулями), Windows Admin Center. 
#### Сценарий №4
Если среда разворачивается на одном хосте то можно использовать паравиртуализацию VirtualBox, VMware Workstation, Hyper-V. Я бы организовывал на Proxmox VE с использованием LXD контейнеризации, под каждый дистрибутив создал отдельную LXD + подготовил docker/docker compose окружение. Далее можно использовать как шаблон LXD для быстрого deploy. В задаче про финансирование не говорится, тогда еще проще, развернул среду в YandexCloud.      

### Ответ задачи №4
##### Недостатки и проблемы гетерогенной среды:
- Разные системы управления, что увеличивает время администрирования;
- Сложная схема автоматизации между гипервизорами;
- Так как технологии различных гипервизоров не совместимы, усложняется и увеличивается время миграции VM;
- Поддержка систем, умение сотрудников управлять в различных системах управления;
- Недостатки интеграция между гипервизорами (backup, совместное использование ресурсов);
- Мониторинг ресурсов, так как используют свои метрики.
##### Для минимизации этих рисков и проблем нужно: 
- Минимизировать гетерогенную среду; 
- Использовать стандарт VM для всех гипервизоров (кол-во CPU, RAM, Disk, Network, OS, протоколы взаимодействия, storage), проще потом администрировать. Стандарт безопасности доступа с использованием Active Directory/LDAP использование централизованной системы безопасности; 
-  Централизованное система резервного копирования (использование продукта, который поддерживает всех участников гетерогенной среды);
- Использование инструментов автоматизации IaaC (конфигурирование, deploy VM, обновление систем), Ansible, Terraform, Puppet. 

> “ Если бы у вас был выбор, создавали бы вы гетерогенную среду или нет? ”

Если бы я очень сильно хотел, то все равно не получилось. Так как есть нюансы, которые возникают в процессе новых задач/целей и функций, в последствии чего, использование той или иной технологии заставляют использовать другую инфраструктуру и среду. 
Просто стемился минимизировать гетерогенную среду, в случае если есть аналоги под существующею инфраструктуру. Использовал однотипные протоколы взаимодействия, централизованное управление и мониторинг, backup итд. 
Вопрос прямой, значит есть выбор на существующею инфраструктуру, то гетерогенную среду исключил.
